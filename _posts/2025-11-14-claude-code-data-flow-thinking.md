Claude Code를 처음 쓰기 시작했을 때 기대는 간단했습니다.
코드를 대신 써주는 도구. "이런 기능 만들어줘"라고 하면 코드가 나오는 것.
그런데 3개월쯤 지나니 이상한 일이 벌어졌습니다.
코드를 짜는 시간은 줄었는데, 생각하는 시간은 오히려 늘었습니다.

## 코딩과 CLI 마스터리는 다른 기술이다

처음엔 혼란스러웠습니다.
Claude Code가 코드를 대신 짜주는데, 왜 아직도 머리가 복잡할까?
답은 단순했습니다. 제가 필요한 건 코딩 능력이 아니었습니다.

CLI 마스터리는 코딩과 다릅니다.
코딩은 새로운 도구를 만드는 것입니다. 함수를 짜고, 클래스를 설계하고, 알고리즘을 구현합니다.
CLI 마스터리는 기존 도구를 엮는 것입니다. 파일을 읽고, 변환하고, 다른 곳에 쓰는 것.

```bash
cat file.txt | grep pattern | sort | uniq > result.txt
```

이 한 줄은 코딩이 아닙니다. 오케스트레이션입니다.
`cat`, `grep`, `sort`, `uniq`은 이미 만들어진 도구입니다.
제가 한 일은 그것들을 파이프(`|`)로 연결한 것뿐입니다.

Claude Code는 이 오케스트레이션을 극대화합니다.
파일을 읽고, Git을 조작하고, API를 호출하고, 결과를 저장합니다.
코드를 짜지 않아도, 시스템 전체를 제어할 수 있습니다.

## 프로그래밍 언어 vs 셸: 서로 다른 표현력

프로그래밍 언어는 메모리와 연산에 특화되어 있습니다.
변수를 선언하고, 반복문을 돌리고, 함수를 호출합니다.
`for`, `while`, `if`로 로직을 표현합니다.

셸은 파일과 프로세스에 특화되어 있습니다.
파이프로 데이터를 흘려보내고, 리다이렉트로 저장하고, 환경변수로 설정을 전달합니다.
`|`, `>`, `<`로 데이터 흐름을 표현합니다.

![기본 데이터 흐름](/assets/images/blog-2025-11-14-data-flow-basic.png)

프로그래밍 언어는 "어떻게 계산할까"에 답합니다.
셸은 "데이터를 어디서 가져와서 어디로 보낼까"에 답합니다.

LLM 시대에는 후자가 더 중요해졌습니다.
계산 로직은 Claude Code가 짜줍니다.
데이터 흐름 설계는 사람이 해야 합니다.

## 데이터 흐름 사고: LLM이 잘하는데 사람이 안 보던 영역

흥미로운 점은 이겁니다.
"데이터 흐름"은 LLM이 못하는 영역이 아니라, LLM이 잘하는 영역인데 인간이 주로 관심을 갖지 않던 영역입니다.

우리는 알고리즘을 배웠습니다. 정렬, 탐색, 동적 프로그래밍.
자료구조를 배웠습니다. 배열, 연결 리스트, 트리, 그래프.
그러나 "파일에서 데이터를 읽어서 변환 후 다른 형식으로 저장"하는 패턴은 별로 안 배웠습니다.

이게 바로 Claude Code가 강력한 이유입니다.
Claude Code는 파일 I/O, JSON 파싱, API 호출, Git 조작을 자연스럽게 연결합니다.
우리가 해야 할 일은 "무엇을 입력으로 받아서 무엇을 출력으로 낼 것인가"를 정의하는 것입니다.

## PARA/CODE 구조: CLI 자동화가 만드는 힘

PARA는 정보 조직 프레임워크입니다.
Projects(프로젝트), Areas(영역), Resources(자료), Archives(보관).
CODE는 정보 처리 프레임워크입니다.
Capture(수집), Organize(정리), Distill(추출), Express(표현).

CODE 레이어에 CLI 자동화가 결합되면 강력해집니다.

예를 들어 클리닉 직원이 "이거 엑셀로 뽑아주세요"라고 요청합니다.
비개발자입니다. 코드를 짤 수 없습니다.
그러나 데이터 흐름은 이해합니다: "시스템에서 데이터를 가져와서 → 엑셀 형식으로 변환 → 파일로 저장"

Claude Code에게 이렇게 말합니다:
"환자 예약 데이터를 가져와서 Excel 파일로 변환해줘"

Claude Code는 다음을 실행합니다:
1. 데이터베이스에서 예약 정보 조회
2. JSON → Excel 변환 스크립트 작성
3. 파일 저장 및 경로 반환

코딩 없이 자동화가 완성됩니다.
필요한 건 데이터 흐름 사고입니다.

## 학습 로드맵: 배선 엔지니어로 사고하기

Claude Code를 효과적으로 쓰려면 무엇을 배워야 할까요?

첫째, Unix 철학을 이해합니다.
"모든 것은 파일이다." "텍스트 스트림으로 통신한다."
이 두 원칙이 CLI의 기초입니다.

둘째, 텍스트 처리 파이프라인을 연습합니다.
`grep`, `sed`, `awk`, `jq`로 데이터를 변환하는 패턴을 익힙니다.
하나의 도구가 다음 도구의 입력이 되는 흐름을 체화합니다.

셋째, OS 기초를 배웁니다.
프로세스, 파일 디스크립터, 환경 변수가 무엇인지 알아야 Claude Code와 대화할 수 있습니다.

넷째, 포맷 변환 도구를 익힙니다.
CSV → JSON → Markdown → HTML.
데이터는 형식만 바뀔 뿐, 본질은 같습니다.

다섯째, LLM을 "배선 엔지니어(wiring engineer)"로 봅니다.
LLM은 도구를 만드는 게 아니라, 기존 도구를 연결하는 전문가입니다.
우리는 "어떤 선을 어디에 연결할지" 설계도를 그립니다.

## 시스템 마스터, 페어 프로그래머가 아니라

Claude Code는 "페어 프로그래머"가 아닙니다.
옆에서 코드를 같이 짜는 동료가 아닙니다.

Claude Code는 "시스템 마스터"입니다.
환경 전체를 제어하는 존재입니다.
파일을 읽고, Git을 조작하고, 워크플로를 실행하고, 알림을 보냅니다.

제가 하는 일은 시스템 설계입니다.
"이 데이터를 저기서 가져와서 이렇게 변환 후 저기에 저장"
이 한 문장이면 충분합니다.

코딩은 Claude Code가 합니다.
데이터 흐름 설계는 제가 합니다.

역할 분담이 명확해지니, 오히려 더 많이 생각하게 되었습니다.
코드 문법 대신, 시스템 구조를 고민합니다.

## 코드는 쉬워졌지만, 설계는 더 어려워졌다

이게 가장 역설적인 부분입니다.
Claude Code가 코드를 대신 써줄수록, 시스템 사고의 중요성은 오히려 커집니다.

코딩 능력이 필요 없어진 게 아닙니다.
코딩 능력이 "주어진 것"이 되었습니다.
그래서 이제 더 본질적인 질문을 던져야 합니다:

- 무엇을 입력으로 받을 것인가?
- 어떤 변환을 거칠 것인가?
- 무엇을 출력으로 낼 것인가?
- 실패하면 어떻게 할 것인가?
- 어떻게 검증할 것인가?

이 질문들은 코드 문법과 무관합니다.
데이터 흐름 설계입니다.
시스템 사고입니다.

Claude Code는 코딩을 쉽게 만들었습니다.
그래서 이제 우리는 설계에 집중할 수 있습니다.
아니, 집중해야 합니다.

## 실패 케이스

이 접근이 작동하지 않는 경우도 있습니다.

첫째, 데이터 흐름이 10단계를 넘어가면 복잡도가 폭발합니다.
중간 단계를 모듈화하지 않으면 추적이 불가능합니다.

둘째, 순환 참조가 생기면 설계 오류입니다.
A → B → C → A 같은 흐름은 무한루프를 만듭니다.

셋째, 외부 API 타임아웃은 제어할 수 없습니다.
네트워크 오류, 서비스 장애는 데이터 흐름 설계로 해결되지 않습니다.

그래도 80%는 해결됩니다.
나머지 20%는 아직 모릅니다.
